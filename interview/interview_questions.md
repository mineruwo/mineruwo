<details>
<summary>Spring Framework와 Spring Boot의 차이점은 무엇인가요?</summary>
<p>

- **Spring Framework**: 자바 개발을 위한 포괄적인 프로그래밍 및 구성 모델을 제공하는 프레임워크입니다. 의존성 주입(DI), 관점 지향 프로그래밍(AOP) 등 핵심 기능을 제공하지만, 애플리케이션 설정을 개발자가 직접 XML이나 Java 파일을 통해 상세하게 해야 하는 번거로움이 있습니다.
- **Spring Boot**: Spring Framework를 더 쉽고 빠르게 사용할 수 있도록 만들어진 도구입니다. '설정보다 관례(Convention over Configuration)' 철학을 바탕으로, 자주 사용되는 설정들을 기본적으로 내장하고 있습니다(Auto-configuration). 내장 WAS(Tomcat 등)를 포함하여 단독으로 실행 가능한 JAR 파일을 쉽게 생성할 수 있고, 스타터(starter) 의존성을 통해 빌드 구성을 간소화해줍니다. 즉, Spring Boot는 Spring Framework를 기반으로 최소한의 설정으로 빠르게 프로덕션 수준의 애플리케이션을 만들 수 있게 해주는 '발사대'와 같습니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
이 답변은 Spring의 공식 문서와 핵심 철학에 기반합니다. Spring Boot의 핵심 가치는 "설정보다 관례(Convention over Configuration)", "자동 설정(Auto-configuration)", "스타터 의존성(Starter Dependencies)"에 있으며, 이는 개발자가 복잡한 설정 없이 빠르게 애플리케이션을 실행할 수 있도록("Just Run") 돕는 것을 목표로 합니다.
</p>
</details>

<details>
<summary>JWT(JSON Web Token)의 구조와 이를 이용한 인증/인가 처리 흐름에 대해 설명해주세요.</summary>
<p>

- **구조**: JWT는 `.`으로 구분된 세 부분으로 구성됩니다: **Header**, **Payload**, **Signature**.
  1.  **Header**: 토큰의 타입(JWT)과 서명에 사용된 해싱 알고리즘(e.g., HS256) 정보를 담습니다. Base64로 인코딩됩니다.
  2.  **Payload**: 토큰에 담을 정보(Claim)를 포함합니다. 사용자의 ID, 이름, 만료 시간 등을 담으며, Base64로 인코딩됩니다.
  3.  **Signature**: Header와 Payload를 Base64 인코딩한 값과 서버가 가진 비밀 키(Secret Key)를 Header에 명시된 알고리즘으로 해싱하여 생성합니다. 토큰의 위변조 여부를 검증하는 데 사용됩니다.

- **인증/인가 흐름**:
  1.  **로그인**: 사용자가 ID/PW로 로그인을 요청합니다.
  2.  **토큰 발급**: 서버는 사용자 정보를 확인하고, 인증이 완료되면 JWT(Access Token, Refresh Token)를 생성하여 사용자에게 발급합니다.
  3.  **요청 시 토큰 포함**: 클라이언트는 발급받은 Access Token을 저장해두고, API 요청 시마다 HTTP Header(주로 `Authorization: Bearer <token>`)에 담아 서버로 전송합니다.
  4.  **서버 검증**: 서버는 요청 헤더의 토큰을 받아 Signature를 검증하여 토큰의 유효성(위변조 여부, 만료 시간)을 확인합니다.
  5.  **인가 처리**: 검증이 성공하면, 서버는 Payload에 담긴 사용자 정보를 기반으로 요청에 대한 인가(권한)를 확인하고, 요청을 처리한 후 응답합니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
이 답변은 JWT의 표준 명세인 **RFC 7519**에 정의된 내용을 기반으로 합니다. Header, Payload, Signature로 구성된 3부분 구조는 이 표준에 명시되어 있습니다. 또한, 여기서 설명하는 인증 흐름은 서버가 상태를 저장하지 않는 '상태-없음(Stateless)' 특징을 가지는 토큰 기반 인증(Token-Based Authentication)의 표준적인 구현 방식입니다.
</p>
</details>

<details>
<summary>좋은 REST API를 설계하기 위한 원칙은 무엇이라고 생각하시나요?</summary>
<p>

(URI, HTTP Method, HTTP Status Code를 중심으로)
1.  **자원(Resource) 중심의 URI 설계**: URI는 동사가 아닌 명사를 사용하여 '자원'을 표현해야 합니다. (e.g., `GET /users/1` (O), `GET /getUser/1` (X)). 자원의 계층 구조는 `/`로 표현합니다.
2.  **행위(Verb)는 HTTP Method로 표현**: 자원에 대한 행위는 HTTP Method(GET, POST, PUT, PATCH, DELETE)를 통해 명확하게 표현해야 합니다.
    - `GET`: 자원 조회
    - `POST`: 자원 생성
    - `PUT`: 자원 전체 교체
    - `PATCH`: 자원 일부 수정
    - `DELETE`: 자원 삭제
3.  **정확한 상태 코드(Status Code) 반환**: 요청의 처리 결과를 명확하게 나타내는 HTTP 상태 코드를 반환해야 합니다. (e.g., `200 OK`, `201 Created`, `400 Bad Request`, `404 Not Found`, `500 Internal Server Error`).
4.  **일관된 컨벤션 유지**: URI의 명명 규칙(e.g., 소문자, 하이픈 사용), 페이로드의 JSON 구조 등 일관된 규칙을 유지하여 API의 예측 가능성과 사용성을 높여야 합니다.
5.  **HATEOAS (Hypermedia as the Engine of Application State)**: (심화) 응답에 현재 리소스와 관련된 다른 리소스로의 하이퍼링크를 포함시켜, 클라이언트가 API의 상태 변화에 따라 동적으로 상호작용할 수 있도록 설계합니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
이 원칙들은 REST(Representational State Transfer) 아키텍처 스타일을 처음 정의한 **로이 필딩(Roy Fielding)의 박사 논문**에서 파생되었습니다. 특히 답변에서 설명하는 내용은 REST의 핵심 제약 조건 중 하나인 '균일 인터페이스(Uniform Interface)'를 실제로 구현하는 방법에 해당합니다.
</p>
</details>

<details>
<summary>JPA(Java Persistence API)를 사용하면서 겪을 수 있는 N+1 문제는 무엇이고, 어떻게 해결할 수 있나요?</summary>
<p>

- **N+1 문제란?** 연관 관계가 설정된 엔티티를 조회할 때, 첫 번째 쿼리(1)로 기본 엔티티 목록을 가져온 후, 해당 엔티티와 연관된 다른 엔티티를 조회하기 위해 N개의 추가 쿼리가 발생하는 문제입니다. (e.g., 1개의 게시글 목록 쿼리 후, 각 게시글의 댓글을 가져오기 위해 N개의 댓글 쿼리가 추가로 실행되는 상황). 이는 데이터베이스에 심각한 부하를 줄 수 있습니다.
- **해결 방안**:
  1.  **Fetch Join**: JPQL에서 `JOIN FETCH` 키워드를 사용하여, 조회하는 주 엔티티와 연관된 엔티티를 즉시 함께 조회하는 방법입니다. 가장 일반적이고 효과적인 해결책입니다.
  2.  **@EntityGraph**: JPA 2.1부터 지원하는 기능으로, 조회 시점에 함께 가져올 연관 엔티티를 동적으로 지정할 수 있습니다.
  3.  **Batch Size**: `@BatchSize` 어노테이션이나 `hibernate.default_batch_fetch_size` 옵션을 사용하여, 지연 로딩 시 한 번에 지정된 개수(N)만큼의 엔티티를 IN 절을 통해 묶어서 조회하는 방법입니다. Fetch Join보다 유연하게 적용 가능합니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
N+1 문제는 ORM(Object-Relational Mapping) 프레임워크에서 객체 그래프와 관계형 데이터 모델 간의 불일치로 인해 발생하는 잘 알려진 성능 문제입니다. ORM이 지연 로딩(Lazy Loading)을 수행할 때, 연관된 컬렉션을 실제로 사용하는 시점에 추가 쿼리를 실행하기 때문에 발생합니다. Fetch Join, EntityGraph, Batch Size는 JPA/Hibernate 공식 문서에서 이러한 문제를 해결하기 위해 제공하는 표준적인 기능이자 패턴입니다.
</p>
</details>

<details>
<summary>JUnit으로 단위 테스트를 진행하셨는데, 단위 테스트의 중요성은 무엇이며, 좋은 단위 테스트는 어떻게 작성해야 할까요?</summary>
<p>

- **중요성**:
  1.  **코드 안정성 확보**: 새로운 기능 추가나 리팩토링 시, 기존 코드의 동작이 깨지지 않았음을 보장하여 심리적 안정감을 줍니다.
  2.  **빠른 피드백**: 개발 과정에서 버그를 조기에 발견하고 수정할 수 있게 해줍니다.
  3.  **살아있는 문서**: 테스트 코드는 해당 코드의 기능과 사용법을 설명하는 가장 정확한 문서 역할을 합니다.
  4.  **설계 개선**: 테스트하기 어려운 코드는 보통 설계가 잘못되었을 가능성이 높습니다. 테스트 작성을 통해 자연스럽게 더 나은 설계(낮은 결합도, 높은 응집도)를 유도할 수 있습니다.
- **좋은 단위 테스트 (FIRST 원칙)**:
    - **Fast**: 테스트는 빠르게 실행되어야 합니다.
    - **Independent**: 각 테스트는 서로 독립적이어야 하며, 실행 순서에 의존해서는 안 됩니다.
    - **Repeatable**: 어떤 환경에서도 반복적으로 동일한 결과를 내야 합니다.
    - **Self-Validating**: 테스트는 성공 또는 실패를 스스로 판단해야 하며, 수동적인 확인이 필요 없습니다.
    - **Timely**: 테스트는 테스트하려는 프로덕션 코드와 비슷한 시점에 작성되어야 합니다 (TDD).

</p>
<hr>
<p>
<strong>근거</strong><br>
테스트의 중요성은 애자일(Agile) 및 테스트 주도 개발(TDD) 방법론에서 강조하는 핵심적인 소프트웨어 공학 원칙입니다. 좋은 단위 테스트의 조건으로 제시된 **FIRST** 원칙은 로버트 C. 마틴(Robert C. Martin)의 저서 "클린 코드(Clean Code)" 등을 통해 널리 알려진 업계 표준 가이드라인입니다.
</p>
</details>

**Frontend (React / JavaScript)**

<details>
<summary>React의 가상 DOM(Virtual DOM)이 왜 필요한지, 어떻게 동작하는지 설명해주세요.</summary>
<p>

- **필요성**: 실제 DOM(Browser DOM) 조작은 비용이 많이 드는 작업입니다. UI에 작은 변화가 생길 때마다 실제 DOM을 직접 수정하면, 전체 렌더링 과정이 반복되어 성능 저하가 발생할 수 있습니다. 가상 DOM은 이러한 비효율을 해결하기 위해 등장했습니다.
- **동작 방식**:
  1.  **메모리 상의 표현**: 가상 DOM은 실제 DOM의 구조를 흉내 내는 간단한 JavaScript 객체입니다. 메모리에 존재하므로 조작 비용이 저렴합니다.
  2.  **변경 감지**: 데이터(state, props)가 변경되면, React는 새로운 가상 DOM 트리를 생성합니다.
  3.  **비교 (Diffing)**: 이전 가상 DOM 트리와 새로운 가상 DOM 트리를 비교하여 변경된 부분만 찾아냅니다 (이 과정을 '재조정(Reconciliation)'이라고 합니다).
  4.  **일괄 업데이트 (Batch Update)**: 변경된 부분들을 모아서 한 번에 실제 DOM에 적용합니다. 이를 통해 불필요한 DOM 조작을 최소화하고 성능을 향상시킵니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
가상 DOM은 React의 핵심 아키텍처이며, 공식 문서의 '재조정(Reconciliation)' 파트에서 그 동작 원리를 설명하고 있습니다. 성능 향상의 원리는 브라우저의 DOM API를 직접, 그리고 반복적으로 호출하는 것을 최소화하는 데 있습니다. DOM API 호출은 브라우저의 리플로우(reflow)와 리페인트(repaint)를 유발하는 비싼 작업이기 때문입니다. React의 선언적 접근 방식과 가상 DOM의 'Diffing' 알고리즘은 이 과정을 최적화합니다.
</p>
</details>

<details>
<summary>React의 클래스 컴포넌트와 함수형 컴포넌트의 차이점은 무엇인가요? React Hooks는 왜 등장했나요?</summary>
<p>

- **차이점**:
    - **클래스 컴포넌트**: `React.Component`를 상속받는 ES6 클래스입니다. `state` 관리, `LifeCycle` 메서드 사용이 가능하며, `render()` 메서드에서 JSX를 반환합니다. 코드가 비교적 장황하고, `this` 바인딩의 복잡성이 있습니다.
    - **함수형 컴포넌트**: 간단한 JavaScript 함수입니다. 과거에는 `state`나 `LifeCycle` 관리가 불가능하여 주로 UI 렌더링만 담당했습니다.
- **Hooks의 등장**: 함수형 컴포넌트에서도 클래스 컴포넌트의 기능(state, lifecycle 등)을 사용할 수 있게 해주는 기능입니다.
    - **등장 배경**:
        1.  **로직 재사용의 어려움**: 클래스 컴포넌트에서는 HOC(Higher-Order Components)나 Render Props 같은 복잡한 패턴을 사용해야 로직 재사용이 가능했습니다.
        2.  **거대한 컴포넌트**: LifeCycle 메서드(e.g., `componentDidMount`, `componentDidUpdate`) 안에 관련 없는 로직들이 섞여 컴포넌트가 비대해지고 이해하기 어려워졌습니다.
        3.  **클래스의 혼란**: `this` 키워드, 이벤트 핸들러의 바인딩 등 클래스 문법 자체가 주는 혼란 때문입니다.
    - **Hooks의 장점**: `useState`, `useEffect` 등의 Hook을 통해 함수형 컴포넌트에서 상태 관리와 생명주기 관리를 더 직관적이고 간결하게 할 수 있으며, 커스텀 Hook을 통해 로직을 쉽게 재사용할 수 있습니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
이 내용은 React의 역사와 발전에 대한 것으로, React 공식 문서의 "Hooks 도입(Introducing Hooks)" 섹션에 상세히 설명되어 있습니다. Hooks의 등장은 클래스 컴포넌트가 가졌던 고질적인 문제들(로직 재사용의 복잡성, 거대하고 이해하기 힘든 컴포넌트, `this` 키워드의 혼란 등)을 해결하기 위함이었습니다. Hooks는 관련 있는 로직을 한곳에 모으고(co-location), 커스텀 Hook을 통해 재사용성을 극대화하는 것을 가능하게 했습니다.
</p>
</details>

<details>
<summary>JavaScript의 `let`, `const`, `var`의 차이점과 호이스팅(Hoisting)에 대해 설명해주세요.</summary>
<p>

- **차이점**:
    - **`var`**: **함수 스코프(function-scoped)**. 변수 재선언 및 재할당이 모두 가능합니다. 호이스팅 시 `undefined`로 초기화됩니다.
    - **`let`**: **블록 스코프(block-scoped, `{}` 기준)**. 변수 재선언은 불가능하지만, 재할당은 가능합니다.
    - **`const`**: **블록 스코프(block-scoped)**. 변수 재선언 및 재할당이 모두 불가능합니다. 선언과 동시에 반드시 초기화해야 합니다. (단, 객체나 배열의 내부 값은 변경 가능합니다).
- **호이스팅(Hoisting)**: JavaScript 엔진이 코드를 실행하기 전, 변수 및 함수 선언을 해당 스코프의 최상단으로 끌어올리는 것처럼 동작하는 현상입니다.
    - `var`로 선언된 변수는 선언부가 끌어올려지고 `undefined`로 초기화됩니다. 따라서 선언 전에 접근해도 에러가 나지 않고 `undefined`가 반환됩니다.
    - `let`과 `const`로 선언된 변수도 호이스팅되지만, `var`와 달리 초기화되지 않습니다. 이 때문에 선언 전에 접근하면 **TDZ(Temporal Dead Zone)**에 해당하여 참조 에러(ReferenceError)가 발생합니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
이 내용은 `let`과 `const`를 도입한 **ECMAScript 2015(ES6) 명세**에 기반합니다. `var`의 함수 스코프와 `let`/`const`의 블록 스코프는 명세에 정의된 가장 큰 차이점입니다. 또한, `let`/`const` 변수에 선언 전에 접근할 수 없는 현상인 **TDZ(Temporal Dead Zone)** 역시 `var`의 비직관적인 동작을 보완하고 코드의 안정성을 높이기 위해 ES6 명세에 포함된 개념입니다.
</p>
</details>

<details>
<summary>`Promise`와 `async/await`는 비동기 처리를 위해 어떻게 사용되는지 설명해주세요.</summary>
<p>

- **`Promise`**: 비동기 작업의 최종 성공 또는 실패를 나타내는 객체입니다. `pending`(대기), `fulfilled`(성공), `rejected`(실패) 세 가지 상태를 가집니다. `.then()`으로 성공 시의 로직을, `.catch()`로 실패 시의 로직을 체이닝(chaining)하여 콜백 지옥(Callback Hell)을 해결합니다.
- **`async/await`**: `Promise`를 더 동기적인 코드처럼 보이게 만들어 가독성을 높이는 문법적 설탕(Syntactic Sugar)입니다.
    - `async` 키워드는 함수 앞에 붙이며, 해당 함수가 항상 `Promise`를 반환하도록 합니다.
    - `await` 키워드는 `async` 함수 내에서만 사용 가능하며, `Promise`가 처리될 때까지(fulfilled 또는 rejected) 함수의 실행을 일시 중지하고 기다립니다. `try...catch` 구문을 사용하여 에러 처리를 동기 코드처럼 할 수 있습니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
`Promise`는 **ES6(ECMAScript 2015)**에서 비동기 처리를 표준화하기 위해 도입되었으며, `async/await`는 **ES8(ECMAScript 2017)**에서 `Promise`를 더 쉽게 사용하기 위해 추가된 문법입니다. 이들은 모두 ECMAScript 표준 명세에 포함된 기능으로, 기존의 콜백 패턴이 가진 가독성 문제(콜백 지옥)를 해결하고 비동기 코드를 동기 코드처럼 작성할 수 있게 하여 코드의 유지보수성을 높이는 것을 목표로 합니다.
</p>
</details>

<details>
<summary>(프로젝트 연계) `localStorage`의 장단점은 무엇인가요? 다른 클라이언트 사이드 저장소(Session Storage, Cookie)와 비교해서 설명해주세요.</summary>
<p>

- **`localStorage`의 장단점**:
    - **장점**: 데이터가 사용자의 브라우저에 영구적으로 저장되어 탭이나 창을 닫아도 유지됩니다. 저장 공간이 쿠키보다 큽니다(약 5MB). 
    - **단점**: 문자열 데이터만 저장할 수 있습니다. 보안에 취약하여 중요한 정보를 저장하기에 부적합합니다. 모든 탭/창에서 동일한 `localStorage` 객체에 접근하므로 데이터 충돌이 발생할 수 있습니다.
- **다른 저장소와의 비교**:
    - **`Session Storage`**: `localStorage`와 유사하지만, 데이터가 세션(탭/창) 단위로 저장되어 탭이나 창을 닫으면 데이터가 사라집니다.
    - **`Cookie`**: 저장 공간이 매우 작고(약 4KB), 모든 HTTP 요청에 자동으로 포함되어 서버로 전송되므로 성능에 영향을 줄 수 있습니다. 주로 인증 상태 유지 등 서버와의 통신에 필요한 최소한의 정보를 저장하는 데 사용됩니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
이 내용은 W3C에서 정의한 **웹 스토리지(Web Storage) API 명세**와 쿠키의 표준인 **RFC 6265**에 기반합니다. 각 저장소의 데이터 생명 주기(영구, 세션), 저장 용량, 서버 전송 여부, 보안 특성은 이 표준들에서 명확하게 정의하고 있는 핵심적인 차이점입니다.
</p>
</details>

<summary>데이터베이스 인덱스(Index)는 무엇이고 왜 사용하나요? 인덱스의 장단점에 대해 설명해주세요.</summary>
<p>

- **인덱스란?** 데이터베이스 테이블의 검색(SELECT) 속도를 향상시키기 위한 자료구조입니다. 책의 '찾아보기'와 같은 역할을 합니다.
- **사용 이유**: 인덱스가 없으면 특정 데이터를 찾기 위해 테이블 전체를 스캔(Full Table Scan)해야 하므로 데이터가 많아질수록 성능이 급격히 저하됩니다. 인덱스를 사용하면 데이터를 빠르게 찾을 수 있어 검색 성능이 크게 향상됩니다.
- **장점**:
    - `WHERE`절이나 `JOIN`을 사용한 검색 쿼리의 성능을 크게 향상시킵니다.
    - 정렬(ORDER BY) 작업의 성능을 향상시킬 수 있습니다.
- **단점**:
    - `INSERT`, `UPDATE`, `DELETE` 작업 시 인덱스 테이블도 함께 수정해야 하므로 쓰기(Write) 성능이 저하됩니다.
    - 인덱스를 저장하기 위한 추가적인 저장 공간이 필요합니다.
    - 인덱스를 잘못 설계하면 오히려 성능이 저하될 수 있습니다 (e.g., 카디널리티가 낮은 컬럼에 인덱스를 거는 경우).

</p>
<hr>
<p>
<strong>근거</strong><br>
인덱스는 데이터베이스 시스템의 근간을 이루는 개념입니다. 대부분의 관계형 데이터베이스는 인덱스를 구현하기 위해 **B-Tree(또는 B+ Tree) 자료구조**를 사용합니다. B-Tree는 데이터가 정렬된 상태를 유지하며, 데이터의 검색, 삽입, 삭제를 모두 로그 시간(O(log n))에 가깝게 처리할 수 있어 대용량 데이터의 빠른 검색을 가능하게 합니다. 읽기(검색) 성능과 쓰기(수정) 성능 간의 트레이드오프는 B-Tree 구조를 유지하기 위한 비용에서 비롯되는 필연적인 특징입니다.
</p>
</details>

<details>
<summary>트랜잭션(Transaction)이란 무엇이며, ACID 원칙에 대해 설명해주세요.</summary>
<p>

- **트랜잭션이란?** 데이터베이스의 상태를 변화시키기 위해 수행하는 논리적인 작업 단위입니다. 여러 개의 쿼리를 하나의 묶음으로 처리하여, 모두 성공하거나 모두 실패하도록 보장(All or Nothing)합니다.
- **ACID 원칙**: 트랜잭션이 안전하게 수행되기 위해 보장해야 하는 4가지 성질입니다.
  1.  **원자성(Atomicity)**: 트랜잭션에 포함된 모든 작업이 전부 성공하거나 전부 실패하여 결과를 예측할 수 있어야 합니다.
  2.  **일관성(Consistency)**: 트랜잭션이 성공적으로 완료되면 데이터베이스는 항상 일관된 상태를 유지해야 합니다 (e.g., 데이터 무결성 제약 조건 만족).
  3.  **고립성(Isolation)**: 하나의 트랜잭션이 실행되는 동안 다른 트랜잭션의 영향을 받지 않아야 합니다. 여러 트랜잭션이 동시에 실행될 때, 마치 순차적으로 실행되는 것처럼 보여야 합니다. (격리 수준(Isolation Level)에 따라 정도가 다릅니다).
  4.  **지속성(Durability)**: 성공적으로 완료된 트랜잭션의 결과는 시스템에 장애가 발생하더라도 영구적으로 저장되어야 합니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
트랜잭션과 **ACID 원칙**은 데이터베이스 관리 시스템(DBMS)의 데이터 무결성과 신뢰성을 보장하기 위한 핵심적인 이론입니다. 이 원칙들은 거의 모든 데이터베이스 이론 서적과 공식 문서에서 기본으로 다루며, 동시성 제어(Concurrency Control)와 회복(Recovery) 기법의 이론적 토대가 됩니다.
</p>
</details>

<details>
<summary>좋은 데이터베이스 스키마를 설계하기 위해 가장 중요하게 생각하는 것은 무엇인가요?</summary>
<p>

좋은 데이터베이스 스키마 설계를 위해 가장 중요한 것은 **정규화(Normalization)를 통해 데이터의 중복을 최소화하고 무결성을 유지하는 것**입니다. 

- **중복 최소화**: 데이터 중복을 줄이면 저장 공간을 효율적으로 사용하고, 데이터 수정 시 발생할 수 있는 이상 현상(Anomaly)을 방지할 수 있습니다.
- **데이터 무결성**: 데이터가 항상 정확하고 일관된 상태를 유지하도록 보장하는 것이 중요합니다. 이를 위해 각 테이블의 목적을 명확히 하고, 적절한 데이터 타입과 제약 조건(기본 키, 외래 키 등)을 설정해야 합니다.
- **확장성 및 유연성**: 향후 요구사항 변경에 유연하게 대처할 수 있도록 스키마를 설계해야 합니다. 

이러한 원칙들을 균형 있게 고려하여, 비즈니스 요구사항에 맞는 최적의 스키마를 설계하는 것이 중요합니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
이 답변은 관계형 데이터베이스 설계의 핵심 원리인 **정규화(Normalization) 이론**에 기반합니다. 정규화는 데이터의 중복성을 제거하여 데이터 무결성을 향상시키고, 삽입/갱신/삭제 이상(Anomaly)을 방지하는 것을 목표로 하는 체계적인 프로세스입니다. 제1정규형부터 제5정규형까지 있으며, 보통 제3정규형이나 보이스-코드 정규형(BCNF)까지 만족시키는 것을 목표로 합니다. 때로는 성능을 위해 의도적으로 정규화를 위배하는 반정규화(Denormalization)를 수행하기도 합니다.
</p>
</details>

<summary>Docker를 사용했을 때의 가장 큰 이점은 무엇이었나요?</summary>
<p>

1.  **환경 격리 및 일관성**: 개발, 테스트, 프로덕션 환경을 컨테이너를 통해 동일하게 구성할 수 있습니다. 이를 통해 "제 PC에서는 됐는데..." 하는 문제를 원천적으로 차단할 수 있습니다.
2.  **빠른 배포 및 확장**: 가상 머신(VM)보다 훨씬 가볍고 빠르게 시작됩니다. 이를 통해 애플리케이션을 신속하게 배포하고, 필요에 따라 수평적으로 확장(Scale-out)하기 용이합니다.
3.  **의존성 관리 용이**: 애플리케이션 실행에 필요한 모든 라이브러리와 의존성을 컨테이너 이미지에 포함시킬 수 있어, 호스트 시스템의 상태와 무관하게 안정적인 실행을 보장합니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
이 답변은 컨테이너 기술의 핵심 가치에 기반합니다. Docker는 **OS 수준 가상화** 기술을 사용하여 호스트 OS의 커널을 공유하면서 프로세스를 격리합니다. 이는 하드웨어 전체를 가상화하는 VM(가상 머신) 방식보다 훨씬 가볍고 빠릅니다. `Dockerfile`과 컨테이너 이미지를 통해 애플리케이션과 그 의존성을 하나의 패키지로 묶어 이식성과 재현성을 보장하는 것이 Docker의 핵심적인 장점입니다.
</p>
</details>

<details>
<summary>AWS RDS와 S3는 각각 어떤 서비스인지 간략하게 설명해주세요.</summary>
<p>

- **AWS RDS (Relational Database Service)**: AWS에서 제공하는 관리형 관계형 데이터베이스 서비스입니다. 사용자가 직접 데이터베이스 서버를 설치, 설정, 관리(패치, 백업 등)하는 부담을 덜어줍니다. MySQL, PostgreSQL, Oracle 등 다양한 데이터베이스 엔진을 지원합니다.
- **AWS S3 (Simple Storage Service)**: AWS에서 제공하는 객체 스토리지(Object Storage) 서비스입니다. 이미지, 동영상, 로그, 백업 데이터 등 모든 종류의 데이터를 사실상 무한대로 저장할 수 있습니다. 높은 내구성과 가용성을 제공하며, 정적 웹사이트 호스팅 등 다양한 용도로 활용됩니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
이 답변은 AWS의 공식 서비스 설명에 기반합니다. RDS는 데이터베이스 관리에 드는 운영 부담을 줄여주는 **PaaS(Platform as a Service)**의 일종이며, S3는 파일 시스템과 달리 계층 구조 없이 객체 단위로 데이터를 저장하는 **객체 스토리지(Object Storage)**의 대표적인 예입니다. 각 서비스의 목적과 사용 사례는 AWS 공식 문서에 명확히 정의되어 있습니다.
</p>
</details>

<details>
<summary>Render와 같은 PaaS 플랫폼을 사용한 배포와, Docker 컨테이너 기반 배포의 차이점은 무엇이라고 생각하시나요?</summary>
<p>

- **PaaS (Platform as a Service, e.g., Render)**: 
    - **장점**: 소스 코드만 푸시하면 빌드, 배포, 인프라 관리가 대부분 자동화되어 매우 편리합니다. 개발자는 인프라 걱정 없이 애플리케이션 개발에만 집중할 수 있습니다.
    - **단점**: 플랫폼이 제공하는 환경에 종속적이며, 세밀한 인프라 제어나 커스터마이징에 한계가 있습니다.
- **Docker 컨테이너 기반 배포**:
    - **장점**: 특정 플랫폼에 종속되지 않아 이식성이 높습니다(AWS, GCP, Azure, 온프레미스 등 어디서든 실행 가능). 운영체제 수준까지 원하는 대로 환경을 구성할 수 있어 유연성과 제어권이 높습니다.
    - **단점**: Dockerfile 작성, 컨테이너 오케스트레이션(필요 시) 등 PaaS에 비해 직접 설정하고 관리해야 할 부분이 많습니다.

**핵심 차이**: PaaS는 **편의성**에, Docker는 **유연성과 이식성**에 더 큰 강점을 가집니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
이 답변은 클라우드 컴퓨팅의 서비스 모델(IaaS, PaaS, SaaS)에 대한 이해를 바탕으로 합니다. PaaS는 개발자가 인프라를 신경 쓰지 않도록 플랫폼을 제공하는 모델이며, Docker와 같은 컨테이너 기술은 그보다 낮은 수준의 인프라(IaaS)를 더 효율적으로 관리하고 애플리케이션을 패키징하는 데 사용됩니다. 즉, **관리의 책임과 제어 수준**에서 가장 큰 차이가 발생합니다. PaaS는 책임이 적고 편리한 대신 제어권이 낮고, Docker는 책임이 많은 대신 제어권이 높습니다.
</p>
</details>

<details>
<summary>사용했던 Git 브랜치 전략(예: Git-flow)에 대해 설명하고, 그 전략을 선택한 이유를 말씀해주세요.</summary>
<p>

- **Git-flow 전략**: 5가지 종류의 브랜치를 사용하여 프로젝트를 관리하는 전략입니다.
    - `main` (또는 `master`): 배포 가능한 프로덕션 코드를 모아두는 브랜치입니다.
    - `develop`: 다음 버전 개발을 위한 코드를 모아두는 브랜치입니다.
    - `feature`: 새로운 기능 개발을 위한 브랜치입니다. `develop`에서 분기하여 개발 완료 후 다시 `develop`으로 병합합니다.
    - `release`: 다음 버전 배포를 준비하기 위한 브랜치입니다. `develop`에서 분기하여 QA, 버그 수정 등을 진행하며, 완료 후 `main`과 `develop`에 모두 병합합니다.
    - `hotfix`: 프로덕션(`main`)에서 발생한 긴급 버그를 수정하기 위한 브랜치입니다. `main`에서 분기하여 수정 후 `main`과 `develop`에 모두 병합합니다.
- **선택 이유**: 역할 분담이 명확하여 여러 개발자가 동시에 협업하는 프로젝트에 적합합니다. 기능 개발, 배포, 긴급 수정 등 각 작업이 독립적인 브랜치에서 이루어지므로 코드 충돌을 최소화하고 안정적으로 프로젝트를 관리할 수 있습니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
Git-flow는 **빈센트 드리센(Vincent Driessen)이 제안한 브랜칭 모델**로, Git의 브랜치 기능을 활용한 워크플로우 패턴입니다. Git 자체의 기능이 아니라, 여러 개발자가 협업하는 환경에서 코드베이스를 체계적으로 관리하기 위해 고안된 널리 알려진 방법론입니다. 이 모델은 명확한 릴리즈 주기가 있는 프로젝트에 특히 효과적입니다.
</p>
</details>

<details>
<summary>HTTP와 HTTPS의 차이점, 그리고 SSL/TLS 핸드셰이크 과정에 대해 설명해주세요.</summary>
<p>

- **HTTP vs HTTPS**:
    - **HTTP (HyperText Transfer Protocol)**: 웹에서 데이터를 주고받기 위한 프로토콜입니다. 텍스트를 암호화하지 않고 그대로 전송하므로, 중간에 데이터를 탈취당할 경우 내용이 그대로 노출되는 보안 취약점이 있습니다.
    - **HTTPS (HTTP Secure)**: HTTP에 SSL/TLS(Secure Sockets Layer/Transport Layer Security) 프로토콜을 추가하여 보안을 강화한 버전입니다. 모든 통신 내용이 암호화되므로, 중간에 데이터를 가로채도 내용을 알 수 없습니다.
- **SSL/TLS 핸드셰이크 과정 (간략히)**:
  1.  **Client Hello**: 클라이언트가 서버에 접속하며, 자신이 사용 가능한 암호화 방식 목록과 랜덤 데이터를 전송합니다.
  2.  **Server Hello**: 서버는 클라이언트의 암호화 방식 중 하나를 선택하고, 자신의 SSL 인증서와 랜덤 데이터를 클라이언트로 전송합니다.
  3.  **인증서 검증 및 키 교환**: 클라이언트는 서버로부터 받은 인증서가 신뢰할 수 있는 CA(Certificate Authority)로부터 발급된 것인지 검증합니다. 검증이 완료되면, 앞으로 통신에 사용할 대칭 키(세션 키)를 생성하고, 서버의 공개키로 암호화하여 서버에 전송합니다.
  4.  **암호화 통신 시작**: 서버는 자신의 개인키로 클라이언트가 보낸 암호화된 대칭 키를 복호화합니다. 이제 클라이언트와 서버는 동일한 대칭 키를 공유하게 되었으므로, 이 키를 사용하여 데이터를 암호화하고 복호화하며 안전하게 통신합니다.

</p>
<hr>
<p>
<strong>근거</strong><br>
이 내용은 HTTP(RFC 2616 등)와 TLS(최신 버전은 RFC 8446)의 공식 기술 명세(RFC)에 기반합니다. HTTPS는 HTTP 통신을 TLS 프로토콜 위에서 수행하는 것을 의미합니다. SSL/TLS 핸드셰이크 과정은 안전한 통신 채널을 수립하기 위한 핵심 절차로, **비대칭키 암호화 방식(공개키/개인키)**으로 상대방을 인증하고 통신에 사용할 **대칭키(세션키)를 안전하게 교환**한 후, 실제 데이터는 연산 비용이 저렴한 **대칭키 암호화 방식**으로 주고받는 원리에 기반합니다.
</p>
</details>
