

# 기술 질문 

## (1) 프론트엔드

### Q1. React를 선택한 이유는 무엇인가요?

**답변:** React는 컴포넌트 기반 구조와 Virtual DOM 덕분에 재사용성이 높고 성능 관리가 용이합니다. 또한 생태계가 커서 라이브러리 활용이 수월하고 커뮤니티 지원도 활발합니다.
**꼬리 질문:**

* Virtual DOM이 실제 DOM보다 무조건 빠른가요?
* React의 단점은 무엇이라고 생각하나요?
* Vue나 Angular를 고려하지 않은 이유는 무엇인가요?

---

### Q2. 상태 관리는 어떤 방식을 사용했나요?

**답변:** 작은 규모에서는 Context API를, 복잡한 전역 상태가 필요할 때는 Redux를 사용했습니다. 프로젝트 성격에 맞게 선택해 불필요한 복잡도를 줄이는 걸 중요하게 생각합니다.
**꼬리 질문:**

* Redux와 Context API의 차이점은 무엇인가요?
* Zustand 같은 다른 상태 관리 도구는 고려했나요?
* 성능 최적화를 위해 어떤 방법을 썼나요?

---

### Q3. 성능 최적화를 위해 어떤 작업을 했나요?

**답변:** React에서는 불필요한 리렌더링을 막기 위해 `memo`, `useMemo`, `useCallback`을 적절히 활용했습니다. 또한 코드 스플리팅과 Lazy Loading을 적용해 초기 로딩 속도를 개선했습니다.
**꼬리 질문:**

* React에서 key 속성을 왜 사용하나요?
* 코드 스플리팅을 적용할 때 주의해야 할 점은 무엇인가요?
* 성능 측정은 어떤 툴로 했나요?

---

## (2) 백엔드

### Q4. Express를 사용한 이유는 무엇인가요?

**답변:** Express는 가볍고 유연하여 REST API 서버를 빠르게 구축할 수 있습니다. 커뮤니티와 미들웨어 생태계도 풍부해서 다양한 요구사항에 대응하기 쉬웠습니다.
**꼬리 질문:**

* NestJS 같은 프레임워크와 비교했을 때 차이점은 무엇인가요?
* Express의 단점은 무엇인가요?
* 에러 처리는 어떻게 구현했나요?

---

### Q5. 인증/인가를 어떻게 구현했나요?

**답변:** JWT 기반 인증을 사용했습니다. 로그인 시 토큰을 발급하고, 이후 요청 시 헤더에 토큰을 담아 검증하는 방식입니다. 필요 시 Refresh Token도 활용했습니다.
**꼬리 질문:**

* JWT와 세션 기반 인증의 차이는 무엇인가요?
* JWT 사용 시 발생할 수 있는 보안 이슈는 무엇인가요?
* OAuth 같은 방식은 고려했나요?

---

### Q6. 데이터베이스는 어떤 것을 사용했나요?

**답변:** MySQL을 사용했습니다. 관계형 데이터 구조와 SQL을 통한 안정적인 쿼리 작성이 가능했기 때문입니다. 복잡한 조인이나 트랜잭션이 필요한 경우에도 적합했습니다.
**꼬리 질문:**

* NoSQL(MongoDB 등)을 고려하지 않은 이유는 무엇인가요?
* 인덱스를 왜 사용하나요?
* 조인이 많아질 때 성능 문제를 어떻게 해결할 건가요?

---

### Q7. API 통신은 어떤 방식으로 했나요?

**답변:** REST API를 기반으로 JSON 형식으로 주고받았습니다. REST는 직관적이고 표준화된 방식이라 프론트엔드와 협업하기 용이했습니다.
**꼬리 질문:**

* REST와 GraphQL의 차이점은 무엇인가요?
* REST API 설계 시 가장 중요한 원칙은 무엇이라고 생각하나요?
* 에러 응답 처리는 어떻게 설계했나요?

---

## (3) 풀스택/배포

### Q8. 배포 환경은 어떻게 구성했나요?

**답변:** Firebase Hosting을 활용해 프론트엔드를 배포했고, 백엔드는 AWS EC2에서 실행했습니다. 데이터베이스는 RDS를 사용해 안정성을 확보했습니다.
**꼬리 질문:**

* Firebase와 AWS의 차이점은 무엇인가요?
* CI/CD 파이프라인을 구성해본 경험이 있나요?
* 서버 다운 시 어떻게 복구했나요?

---

### Q9. Docker를 사용해본 경험이 있나요?

**답변:** 네, 개발 환경 차이를 줄이기 위해 Docker를 활용했습니다. 컨테이너 기반으로 설정을 공유해 팀원들이 동일한 환경에서 작업할 수 있었습니다.
**꼬리 질문:**

* Docker와 VM의 차이는 무엇인가요?
* Docker Compose는 언제 사용하나요?
* 실제 배포에서 Docker를 쓸 때 주의할 점은 무엇인가요?

---

### Q10. 풀스택 개발자로서 가장 어려웠던 점은 무엇인가요?

**답변:** 프론트와 백엔드를 동시에 관리하다 보니 시간 분배와 우선순위 결정이 어려웠습니다. 하지만 전반적인 흐름을 이해할 수 있어 협업 시 큰 장점이 되었습니다.
**꼬리 질문:**

* 프론트와 백엔드 중 더 자신 있는 분야는 어디인가요?
* 풀스택으로 일할 때 가장 중요한 역량은 무엇이라고 생각하나요?
* 만약 백엔드 성능 이슈가 생기면 어떤 접근 방식을 취하나요?

---

## (4) 협업 & 문제 해결

### Q11. Git 협업은 어떻게 했나요?

**답변:** Git Flow 전략을 기반으로 main, develop 브랜치를 운영하고, 기능별로 feature 브랜치를 만들어 작업했습니다. Pull Request를 통해 코드 리뷰를 진행했습니다.
**꼬리 질문:**

* rebase와 merge의 차이는 무엇인가요?
* 충돌이 발생했을 때 어떻게 해결했나요?
* GitHub Actions 같은 자동화 도구를 써봤나요?

---

### Q12. 프로젝트에서 가장 어려웠던 문제는 무엇이었나요?

**답변:** DB 쿼리 성능 이슈가 있었습니다. 인덱스를 추가하고 N+1 문제를 해결하면서 성능을 개선할 수 있었습니다.
**꼬리 질문:**

* N+1 문제란 무엇인가요?
* 캐싱은 고려하지 않았나요?
* 성능 개선 후 얼마나 빨라졌나요?

---

### Q13. API 문서는 어떻게 관리했나요?

**답변:** Swagger를 사용해 자동화된 문서를 생성했습니다. 프론트엔드와 백엔드가 같은 기준을 공유할 수 있어 협업에 큰 도움이 됐습니다.
**꼬리 질문:**

* Swagger 외에 다른 API 문서화 도구를 아나요?
* API 버전 관리는 어떻게 했나요?
* 문서와 실제 코드가 달라졌을 때 어떻게 관리했나요?

---

### Q14. 테스트 코드를 작성해본 경험이 있나요?

**답변:** 핵심 비즈니스 로직에 대해 Jest와 Supertest를 활용해 단위/통합 테스트를 작성했습니다. 이를 통해 배포 전 안정성을 확보했습니다.
**꼬리 질문:**

* 단위 테스트와 통합 테스트의 차이는 무엇인가요?
* TDD 방식을 적용해본 적이 있나요?
* 테스트 커버리지를 높이는 게 항상 좋은 건가요?

---

### Q15. 프로젝트를 다시 한다면 어떤 기술을 바꾸고 싶나요?

**답변:** 백엔드에서는 Express 대신 NestJS를 써보고 싶습니다. 구조가 명확해 확장성 있는 서비스를 만들기 유리하다고 생각하기 때문입니다.
**꼬리 질문:**

* NestJS의 장점은 무엇이라고 생각하나요?
* 새로운 기술을 선택할 때 고려하는 기준은 무엇인가요?
* 기술 스택을 바꾸는 것이 항상 좋은 선택일까요?
